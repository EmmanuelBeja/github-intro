Github: Getting started.

Prerequisites

  • Internet.
  • A github Account.
  • For windows;gitbash. Download gitbash here

Creating A Repository

  • Create repository in your github Account and call it the name of the project.
  • Add description of project.(optional)
  • Check public(For public repository. Private is paid for.)
  • Initialize with a readme.(Optional) To learn more about styling your readme click here
  • Add .gitignore file (optional)
  • Add license(commonly used license for open source projects is MIT. This is also optional)
Cowabunga! You have your first github repository. Simple right? So now we need to put stuff in it.

Pushing to Github

Method 1

Copy link of the project. It could be in either of the two options bellow.

OR


On your computer open the terminal for linux users. For window users, open gitbash comandline tool.
Ensure it is in the place you want to work your project on. Eg. Desktop
now key in the following commands:

$ git clone clonelinkfromgithub


This will download the repository you created on github into your local machine. Copy files you want to upload in the folder you have cloned. Ensuring your terminal/gitbash console is in the cloned project folder follow the steps bellow. You will do this for each and every file. One at a time. This is something we will do for each file again and again untill we have no more remaining.

$ git add fileyouwantpushed
$ git commit -m "describe what you did"
$ git push -u origin master


Wait...What just happened?
git add index.py
Adds the modified and new (untracked) file in the current directory to the staging area (a.k.a. the index), thus preparing it to be included in the next git commit.
git commit -m "describe what you did"
Record changes to the repository. Note: never let the mssg have a ' in between
git push -u origin master
Push changes from the local master branch to the remote master branch. This might prompt you for you your github user name and password depending on configuration of github on your machine.
Use "git status" incase you are lost and need hints from github

Method 2

In you project(which is in your local machine), open terminal in the folder(for linux users) or gitbash(for windows users).

$ git init
$ git add fileyouwantpushed
$ git commit -m 'message about the changes'
$ git remote add origin clonelinkfromgithub
$ git push -u origin master

git init

This command creates an empty Git repository - basically a .git directory with subdirectories for objects, refs/heads, refs/tags, and template files. An initial HEAD file that references the HEAD of the master branch is also created.

Creating A New branch

In the local project that has been interacting with a respective github repo, open your terminal.

$ git checkout -b newbranchname

And you have a new branch in your local machine. So now you need to add content in it then push it to your github to update the repository. After adding content, do as bellow.

$ git add fileyouwantpushed
$ git commit -m "describe what you did"
$ git push -u origin master

When pushing to github, always ensure you are in the right branch. You can know the branch you are in by typing: git branch
Delete a branch on your local filesystem : git branch -d branchname

Fetch

git fetch
Only downloads new data from a remote repository - but it doesn't integrate any of this new data into your working files. Fetch is great for getting a fresh view on all the things that happened in a remote repository. Due to it's "harmless" nature, you can rest assured: fetch will never manipulate, destroy, or screw up anything. This means you can never fetch often enough.

Pull

git pull
Updates your current HEAD branch with the latest changes from the remote server. This means that pull not only downloads new data; it also directly integrates it into your current working copy files.
This has a couple of consequences:

  • Since "git pull" tries to merge remote changes with your local ones, a so-called "merge conflict" can occur. Check out our in-depth tutorial on How to deal with merge conflicts for more information.
  • Like for many other actions, it's highly recommended to start a "git pull" only with a clean working copy. This means that you should not have any uncommitted local changes before you pull. Use Git's Stash feature to save your local changes temporarily.
